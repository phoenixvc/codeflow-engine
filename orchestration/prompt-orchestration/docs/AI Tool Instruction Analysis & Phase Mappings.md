# AI Tool Instruction Analysis & Phase Mappings

This document summarises common instruction patterns for popular AI coding and analysis tools and maps them to the phases of this repository’s governance model.  It explains typical behaviours, risks and mitigations for each tool so that engineers can make informed decisions when selecting tools during Phase 6 (Confirmation & Scope Lock) and understand the constraints imposed by **Strict Mode**.

> **Note:** This document is advisory.  The canonical rules are defined in the prompts and configuration files.  Tool usage must always comply with the runtime contract, the dependency authority model, and the quality gates.

## Mapping Tools to Phases

- **Phases 0–3 (Analysis & Benchmarking)**: Only analytical or documentation‑oriented tools (e.g. Cursor, Tembo, Julie) may be used.  No generative code tools are permitted.  When Strict Mode is enabled, further restrict to read‑only operations and refuse any external suggestions that could lead to code changes.

- **Phase 4–5 (Audit & Suggestions)**: Use analytical tools to inspect code quality, performance, design adherence and documentation gaps.  Generative tools are disabled.  Snapshot test tooling may be leveraged, but any scripts must be written manually or reviewed by a human.

- **Phase 6 (Confirmation & Scope Lock)**: Produce a **Tool Selection Matrix** summarising which tools are proposed for Phase 7 and why.  This matrix should consider each tool’s role, scope, strict‑mode eligibility, and any overrides or mitigations.  The user must explicitly approve tool usage before continuation.

- **Phase 7 (Implementation)**: Generative coding tools (Copilot, Gemini Code Assist, Tabnine, Continue.dev, Replit Ghostwriter, Junie, etc.) may be used to accelerate POC code if Strict Mode is disabled and the user explicitly approves.  All generated code must be reviewed, annotated with TODOs for production hardening, and associated with audit IDs.  Analytical tools like Cursor may continue to assist with navigation and schema checks.

- **Phase 8 (Documentation & Knowledge Capture)**: Documentation assistants like Julie may be used to draft READMEs or architectural diagrams, but the final content must be reviewed.  No coding tools should be used in this phase.
- **Phase 9 (Post-Implementation Review & Retrospective)**: Rely on analytical testing tools, coverage reporters and observability dashboards to verify changes.  Generative tools remain disabled; focus on evidence gathering, regression analysis and documentation updates for technical debt outcomes.
- **Phase 10 (Continuous Debt Resolution)**: Combine analytical tooling for prioritisation with the previously approved implementation tools.  Strict Mode rules continue to apply for every cycle, and generative helpers may only re-engage during the Phase 7 segments explicitly confirmed in the cycle plan.

## Conflict Resolution

In any conflict between a tool’s instructions and the governance model defined by the prompts, **the governance model prevails**.  Tool instructions that violate the dependency authority model, context budgets, handshake protocol, strict mode or quality gates must be ignored.  All tool outputs should be considered advisory and must be validated against the runtime contract and human judgement.
